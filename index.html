02 · API Endpoint

智能路由：按模型能力择优

读取上游 /models 的 supported_endpoints，决定真正请求哪个路径。

  • Claude 优先级 ：/v1/messages → /responses → /chat/completions
  • OpenAI Chat ：模型不支持 Chat 时自动转 Responses
  • OpenAI Responses ：模型不支持 Responses 时回退 Chat
  • Gemini ：先转 OpenAI Chat，再按能力路由；countTokens 本地估算不走上游
03 · 模型 Cache

两层缓存：能力缓存 + Prompt Cache

模型能力缓存

每账号独立缓存 /models，TTL 5 分钟，用于快速判断支持的 endpoint。

Claude Prompt Cache

代理只转发 cache_control 标记并映射 usage，不落地任何 prompt 内容。

Claude 路径cache_control缓存用量映射兼容性
native /v1/messages保留 type/ttl，删 scope保留 cache_read / cache_creation最好
via /responses转换时不保留cached_tokens → cache_read中等
via /chat/completions转换时不保留cached_tokens → cache_read较弱
只有走原生 /v1/messages 时 Prompt Cache 标记才最可能完整传给上游；转 Responses / Chat 后标记丢失，仅保留 usage 命中统计。

04 · 工具调用

三条路径的工具语义保真度

Native Messages 最完整

原生透传，保留 Anthropic tool 语义与流式事件最完整。

Responses 良好

转 function tool，保留 function_call_output 与错误状态。

Chat Completions 基础

转 OpenAI tools / tool_calls，丢失部分语义。

结论：需要最强工具与缓存能力时，让客户端走原生 Anthropic /v1/messages 入口。
