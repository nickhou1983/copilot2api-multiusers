Q2
Issue 02 — 大图片 / 长历史触发 413

413 Request Entity Too Large：根因是 Business 端点网关层

现象:带图片或长历史会话的请求经 /chat/completions 报 413。报错链路:Claude 本身有单图/单请求图片上限;代理侧不压缩图片,历史图片逐轮重复携带 → 请求体越滚越大 → 撞上 Business 端点网关的 body 大小上限 → 413。 逐级加码直连 individual 端点实测(claude-opus-4.8),再与 Business 端点对照:

Payload结果说明
1 MB✓ 200正常返回
2 MB✓ 200正常返回
3 MB ✕ 400 prompt is too long: 1,209,916 tokens > 1,000,000 maximum
Business 端点 · 大 body ✕ 413 api.business.githubcopilot.com 网关层 body size 上限,请求未到模型服务(阈值未验证,预估约 30MB)
  • Claude 图片硬限制:单图 ≤ 5MB、≤ 8000×8000px;单请求 ≤ 20 张,超过则每张降到 ≤ 2000×2000px;超限上游直接 400/413(建议长边 ≤1568px)。超大图 / 超多图本身即报错,与请求体大小是两条独立失败原因。
  • 413 根因链:代理原样透传、不做图片压缩,多轮对话把历史图片全部重复携带 → body 随轮次线性膨胀 → 撞 Business 网关 body 大小上限 → 413(该阈值未验证,预估约 30MB;individual 端点无此网关限,改撞 400 token 上限)。SDK Session 通道对大小更宽松。
  • 带图另一坑:缺 Copilot-Vision-Request: true 头 → "vision is not enabled…",已补齐。VS Code 会把 27MB 原图压到 ~768px(~150KB)再发 —— 客户端「能发大图」实为先压缩,本代理未做同等压缩故更易超限。

Q3
Issue 03 — 经 /chat/completions 转换后的思考签名问题

Thinking Signature：只有原生透传路由能保住它

是什么
Extended Thinking 开启后,每个 thinking 内容块附带一个不透明的 signature 字段 —— 加密验证凭据,证明该思考块确实由 Claude 生成。多轮对话 / 工具调用回传时必须原样带回,被改动或丢失即校验失败。
流式下的形态
content_block_start(thinking) → thinking_delta × N → signature_delta → content_block_stop → text / tool_use
签名在 thinking 块收尾时以 signature_delta 事件补发 —— 代理必须完整转发这一事件序列。
问题:转换路径丢签名
经 /chat/completions(OpenAI 格式)转换时,thinking 块被折叠为 reasoning / 普通文本,signature 字段在两种协议间没有对应位置,无法往返保真 —— 客户端下一轮把思考内容回传时缺签名,触发上游校验失败。
解决方案
需要思考签名的客户端(Claude Code 等)一律走原生 /v1/messages 透传路由:该路由逐字节透传、与直连等价,签名与 signature_delta 完整保留。/chat/completions 仅供无签名回传需求的 OpenAI 兼容客户端使用。

Issue 06 — SSE 流式断连「三兄弟」(2026-07-17)

同一根因，三种死法：thinking 静默期撞超时

典型现象:发出 message_start + content_block_start(thinking 块)后静默 ~180s,然后断流。三种报错对照:

错误含义发起方
stream read error: context canceled 下游客户端不等了(取消/断开) 客户端 代理已归类为 Warn 级正常运维事件
RST_STREAM; INTERNAL_ERROR; received from peer 上游显式重置 HTTP/2 流(带错误码) 上游 网关/LB 空闲超时的一种表现
stream read error: unexpected EOF 连接被「默默」关掉,流没收尾(无 [DONE]/终止块) 上游或中间层 直接 FIN 断连 —— 同一根因的另一种死法
  • 根因:Opus 4.8 长思考期间上游不产出任何字节,撞中间层(网关 / LB)空闲超时。有的中间层发 RST_STREAM,有的直接断 TCP → unexpected EOF。
  • 代理侧排查结论:逐项检查并发模型(每请求一 goroutine)/ 锁(流式循环不持锁)/ CPU / 内存 —— copilot2api 自身性能瓶颈导致 EOF 的可能性极低;若是进程僵死会全量报错,与零星单请求现象不符。
  • 配套咨询:SSE vs WebSocket 差异、HTTP/2 PING 保活机制(应对静默期的正确姿势)。

Extended Topics — 缓存命中与多账户

缓存：前缀匹配是一切；代理只读不算

命中原理(Claude vs OpenAI)
两家同为前缀匹配:从第一个 token 起逐字节比对,遇到第一处不同即中断。区别仅在断点由谁划 —— Claude 显式 cache_control 断点,OpenAI / Copilot 上游自动前缀缓存。代理从不自行判断命中,只读上游 usage 里的 cached_tokens 并转译给客户端。
实战:一个指纹毁掉全部缓存
Claude Code 默认在 system prompt 最开头插入 attribution 块(含每次可变的 prompt fingerprint)→ 前缀从第一个 token 就不同 → cached_tokens 恒为 0。解法:关掉该设置,让 system 开头恢复稳定,命中率立刻回升。
cache_control.scope 分叉
非 Anthropic 标准字段:直连上游 400,经代理有意剥离后 200 —— 95 项矩阵中两处有意分叉之一,是既定兼容行为而非缺陷。缓存按账号隔离,多账户之间互不污染。
多账户工程化
Account Pool:轮询(round-robin)+ 故障转移(e2b62eb);账户 API Key 自动生成与强制校验;Admin UI 新增上游模型页(610d18e)与 Stats 页缓存命中率列(47152bc),命中率从此可观测。
